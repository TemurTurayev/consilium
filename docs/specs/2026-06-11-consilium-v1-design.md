# Consilium v1 — Design Spec

*Дата: 2026-06-11 (rev 2: Rust-ядро, attached-дирижёр). Статус: на ревью у владельца продукта.*
*Рабочий язык спеки — русский; публичные доки (README, сайт) — английский.*

## 1. Что это

**Consilium** — оркестратор, объединяющий подписочные CLI-агенты (Claude Code, Codex CLI, Gemini CLI) в один «консилиум»: агенты совещаются, анонимно ревьюят друг друга и делят работу так, чтобы не сжигать лимит одной подписки.

Метафора — врачебный консилиум: специалисты разных профилей совещаются над одним пациентом. Слоган: *"Get a second opinion. And a third."*

Лицензия MIT, публичный GitHub-репозиторий.

## 2. Ключевые решения (зафиксированы)

| Решение | Выбор | Отклонено |
|---|---|---|
| Форма v1 | Standalone-движок + локальный веб-UI | Форк Warp (1.4M LOC, кастомные панели требуют правок ядра, спек-гейты на контрибьюции, AGPL); плагин только внутри Claude Code (ограничивает UX) |
| Язык ядра | **Rust** (tokio + axum), single binary | TypeScript-ядро (отклонено: Tauri-десктоп Rust-нативен, Warp-экосистема Rust/rmcp, дистрибуция одним бинарником, доверие аудитории GitHub) |
| Веб-UI | TypeScript + React; типы протокола автогенерятся из Rust (ts-rs) | Дублирование типов вручную |
| Доступ к моделям | Только официальные CLI на подписках пользователя | Прямые API-вызовы (теряется суть: экономика подписок) |
| Дирижёр | **attached** по умолчанию: интерактивная сессия Claude Code через MCP (тратит подписку); detached (`claude -p`, тратит кредит) — для автономных прогонов | PTY-автоматизация интерактивного CLI (серая зона, ровно её Anthropic закрывает — не делаем) |
| Дефолтный режим | `auto` (конвейер из примитивов) | Ручной выбор режима на каждую задачу |
| После v1 | v1.1 Warp-адаптер (OSC 777), затем Tauri-десктоп, расширение MCP-поверхности | — |

## 3. Экономическая модель (критично!)

С **15 июня 2026** Anthropic выводит `claude -p`/Agent SDK из flat-пула подписки:
отдельный месячный кредит — Pro $20, Max 5x **$100**, Max 20x $200 (API-эквивалент в долларах).
**Интерактивный Claude Code остаётся на обычных лимитах подписки.** Codex CLI работает
в рамках лимитов ChatGPT-подписки (5-часовые + недельные окна), Gemini CLI — дневные
квоты Google-аккаунта.

Следствия для архитектуры:

1. **Attached-режим — главный ответ на репрайсинг**: дирижёр живёт в интерактивной сессии Claude Code (подписка, flat) и управляет движком через MCP; headless гоняются только Codex/Gemini. Программный кредит Claude почти не тратится.
2. **Claude — низкочастотный мозг**: планирование, арбитраж, синтез. Дефолтный роутинг не отдаёт ему массовую кодогенерацию (Claude-воркер на Sonnet — осознанный выбор пользователя в конфиге, под контролем quota-модуля).
3. **Codex + Gemini — рабочие лошади**: черновая генерация, ресёрч, рутинные правки.
4. **Модуль quota отслеживает четыре пула раздельно**: Claude programmatic credit ($), Claude interactive (окна), ChatGPT (окна), Gemini (дневные квоты). Видимость трат — headline-фича продукта.
5. Опциональный API-key fallback на провайдера, но дефолт — подписки.

## 4. Архитектура

```
consilium/
├── core/                   # движок — Rust (Cargo workspace)
│   └── src/
│       ├── adapters/       # claude.rs, codex.rs, gemini.rs + trait Adapter
│       ├── orchestrator/   # примитивы: council.rs, conduct.rs, review.rs
│       │   └── auto.rs     # конвейер-композиция примитивов (дефолт)
│       ├── supervisor.rs   # непрерывный наблюдатель
│       ├── sessions.rs     # spawn/lifecycle процессов агентов (tokio::process)
│       ├── quota.rs        # учёт пулов, роутинг по нагрузке
│       ├── config.rs       # consilium.config.json (serde + валидация)
│       ├── server/         # axum: REST + WebSocket на localhost
│       └── mcp.rs          # MCP-сервер (rmcp) — attached-режим дирижёра
├── web/                    # UI (React + Vite + Tailwind, TypeScript)
├── bindings/               # ts-rs: автогенерённые TS-типы протокола
└── docs/
```

Движок — один локальный бинарник (`consilium serve`), UI — статика, которую он раздаёт.
Никаких внешних серверов, телеметрии, аккаунтов. Все данные — локально в `~/.consilium/`
(SQLite через rusqlite: сессии, события, счётчики квот).

Стек ядра: tokio (async, процессы), axum (HTTP/WS), serde/serde_json, rusqlite,
rmcp (официальный Rust MCP SDK — его же использует Warp), ts-rs (генерация TS-типов).

### 4.1 Адаптеры

Каждый адаптер запускает официальный CLI в headless-режиме и нормализует его поток в
единый `AgentEvent`:

| Провайдер | Вызов | Парсинг |
|---|---|---|
| Claude (detached) | `claude -p --output-format stream-json --model <m>` | stream-json события |
| Codex | `codex exec --json -m <m>` | JSONL события |
| Gemini | `gemini -p <prompt> -m <m>` (+ JSON-флаги по версии CLI) | stdout/JSON |

`AgentEvent` (Rust enum + serde, экспорт в TS через ts-rs): `SessionStarted`, `Thinking`,
`ToolCall`, `FileChanged`, `Message`, `Usage` (токены), `Completed`, `Failed`.

Принципы: версия CLI определяется на старте (`--version`), формат-брейки изолируются
внутри адаптера; фикстурные тесты на записанных реальных выводах CLI ловят поломки
без расхода лимитов. Новый провайдер = один новый файл-адаптер, реализующий trait.

### 4.2 Примитивы оркестрации

**`council`** (порт паттерна Карпатого на кодинг-CLI):
1. Вопрос уходит N агентам независимо (параллельно).
2. Ответы анонимизируются (Agent A/B/C) — модели не знают, чьё ревьюят.
3. Каждый агент ревьюит и ранжирует чужие ответы по заданным критериям.
4. Председатель (роль `chairman`) синтезирует финальное решение.
Применение: архитектурные решения, сложные баги, выбор подхода.

**`conduct`** (дирижёр):
1. Роль `conductor` разбивает задачу на самодостаточные подзадачи (каждый промпт
   содержит весь нужный контекст — воркеры не видят чужих сессий и истории).
2. Подзадачи распределяются по воркерам с учётом quota и специализации.
3. Воркеры работают параллельно, каждый в своей сессии в рабочей директории проекта.
4. Дирижёр принимает результат: принять / вернуть на доработку (с конкретикой) / интегрировать.
Правило: воркеры никогда не общаются напрямую, весь контекст — через дирижёра.

**`review`** (перекрёстное ревью):
diff одного агента уходит другому на аудит (баги, безопасность, качество);
при споре — арбитр (роль `chairman`). Самый дешёвый повседневный режим.

### 4.3 Режим `auto` (дефолт)

State machine, композирующая примитивы по фазам:

```
задача → [PLANNING]  council: план/подход (если задача нетривиальна)
       → [EXECUTION] conduct: раздача подзадач воркерам
       →             review: каждая завершённая подзадача
       → [INTEGRATION] conductor собирает, гоняет тесты
       → [DONE]      отчёт пользователю
  (supervisor наблюдает непрерывно на всех фазах)
```

Триаж сложности делает дирижёр первым же вызовом: тривиальная задача минует
PLANNING и уходит одному воркеру (экономия дорогих вызовов). Пользователь может
форсировать конкретный примитив вручную (`/council ...`, `/review ...`).

### 4.4 Режимы дирижёра (экономика подписки)

**attached (дефолт при интерактивной работе).** Дирижёр — живая интерактивная сессия
Claude Code пользователя. Движок (`consilium serve`) выставляет MCP-сервер
(`consilium mcp`, stdio или streamable HTTP) с инструментами: `council_run`,
`delegate_task`, `review_diff`, `worker_status`, `quota_status`. Сессия Claude Code
принимает решения (подписка, flat-лимиты), движок исполняет — гоняет Codex/Gemini
headless и стримит события в веб-UI. Программный кредит Claude не тратится.

**detached (автономный).** Полностью фоновый прогон без открытой сессии: Claude-роли
через `claude -p` — расходуют программный кредит. Для ночных прогонов, CI-сценариев,
запуска из веб-UI без открытого Claude Code.

Веб-UI работает в обоих режимах; в attached он зеркалит происходящее и позволяет
управлять воркерами. Движок помечает каждую Claude-сессию её режимом — quota-модуль
относит расход к правильному пулу.

### 4.5 Supervisor («редко, но метко»)

Отдельная роль-наблюдатель. Экономика: чтение (input) кратно дешевле генерации
(output), поэтому супервизор **много читает, мало пишет**.

- Подписан на event-стримы всех активных сессий; держит rolling-конспект происходящего.
- Дефолтный провайдер — **Gemini** (огромный контекст, самые свободные квоты):
  может держать в голове работу всех агентов одновременно.
- Триггеры вмешательства: воркер зациклился (повтор одних действий), дрейф от
  спеки подзадачи, правки файлов вне scope, повторные падения тестов, молчание
  дольше порога.
- Вмешательство: короткая корректирующая заметка в сессию воркера + флаг в UI.
  При серьёзном конфликте — эскалация дирижёру.
- Чувствительность настраивается (`interventionThreshold: low|medium|high`).

### 4.6 Роли и конфиг

`consilium.config.json` (serde-валидация, дефолты «из коробки»):

```jsonc
{
  "roles": {
    "conductor":  { "provider": "claude", "model": "fable-5", "effort": "high",
                    "mode": "attached" },
    "chairman":   { "provider": "claude", "model": "fable-5", "effort": "high" },
    "workers": [
      { "provider": "codex",  "model": "gpt-5.4" },
      { "provider": "gemini", "model": "gemini-3-pro" },
      { "provider": "claude", "model": "sonnet" }   // для обычного кодинга
    ],
    "reviewer":   { "provider": "codex",  "model": "gpt-5.4" },
    "supervisor": { "provider": "gemini", "model": "gemini-3-pro",
                    "interventionThreshold": "medium" }
  },
  "quota": {
    "claude":  { "programmaticCreditUsd": 100 },   // Max 5x
    "gemini":  { "dailyRequests": 1000 },
    "codex":   {}                                   // окна авто-детектятся по ошибкам
  }
}
```

Любая роль — любой провайдер/модель: «управляющие» роли — топ-модель с высоким
effort, рутина — дешёвые модели. Effort транслируется в флаги соответствующего CLI.

### 4.7 Quota и роутинг

- Счётчики из `Usage`-событий адаптеров пишутся в SQLite, окна — скользящие.
- Для Claude programmatic: конвертация токенов в $ по публичному прайсу API
  (конфигурируемая таблица цен) → видно остаток кредита.
- Роутинг воркеров: скоринг = свободная квота × пригодность к типу задачи;
  при исчерпании пула провайдер исключается до восстановления окна, UI предупреждает.

## 5. Веб-UI (localhost)

Три экрана:

1. **Dashboard** — карточки агентов (статус: working/waiting/idle, текущая подзадача),
   индикаторы здоровья всех четырёх квотных пулов, лента вмешательств супервизора.
2. **Session** — живой лог выбранной сессии (события, tool calls), просмотр diff.
3. **Council** — вкладки независимых ответов, матрица взаимных оценок,
   финальный синтез председателя. Витрина продукта (гифка для README).

WS-протокол: UI подписывается на каналы `session:<id>`, `quota`, `supervisor`.
Тёмная тема по умолчанию.

## 6. Тестирование

- Ядро: `cargo test`, TDD; адаптеры — фикстурные тесты на записанных выводах реальных
  CLI (`fixtures/<provider>/<version>/*.jsonl`) — регрессии форматов без расхода лимитов.
- Оркестратор: unit-тесты с mock-адаптером (trait!) — счастливый путь, отказ воркера,
  исчерпание квоты, вмешательство супервизора.
- Web: vitest на компоненты, Playwright-смоук против движка с mock-адаптерами.
- CI: GitHub Actions (cargo fmt + clippy + test; web lint + build + test) на каждый PR.

## 7. Вне scope v1

Warp-адаптер (v1.1; протокол OSC 777 изучен), Tauri-десктоп, git worktree-изоляция
воркеров (v1: общая рабочая директория, последовательная интеграция дирижёром;
worktrees — v1.2), командные/облачные фичи, телеметрия.

## 8. Риски

| Риск | Митигация |
|---|---|
| Форматы headless-вывода CLI меняются | Изоляция в адаптерах + фикстурные тесты + детект версии |
| Rust медленнее в итерациях, чем TS | Основной объём кода пишет Claude; компилятор ловит классы багов до запуска; mock-trait упрощает тесты |
| $100/мес кредита Claude не хватит | Attached-режим по умолчанию (кредит не тратится); quota-модуль показывает остаток; настраиваемый бюджет-стоп |
| Вендор ужесточит политику headless | Attached-режим уже соответствует духу политики; адаптеры абстрагируют способ запуска; API-key fallback |
| Параллельные воркеры конфликтуют по файлам | v1: дирижёр выдаёт непересекающиеся scope; worktrees в v1.2 |
| `codex` CLI не установлен у пользователя | `consilium doctor`: проверка/установка/логин всех CLI |

## 9. Definition of Done для v1

1. `cargo install consilium` (и brew-формула) + `consilium serve` поднимает движок и UI
   на чистой машине; `consilium doctor` проверяет/чинит наличие и логин трёх CLI.
2. **Attached-режим**: из интерактивного Claude Code через MCP вызываются `council_run`
   и `delegate_task` — работа идёт на подписке, программный кредит не тратится
   (видно в quota-панели).
3. Режим `auto` проходит реальную задачу уровня «добавь фичу в небольшой репозиторий»
   end-to-end с видимой работой всех трёх провайдеров.
4. `council` выдаёт синтезированный ответ с матрицей оценок на реальном вопросе.
5. Квоты всех пулов видны и обновляются в реальном времени.
6. Тесты зелёные (cargo + web), CI настроен.
7. README (EN) с гифкой Council-экрана + Quick Start.
